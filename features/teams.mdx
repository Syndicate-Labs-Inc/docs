Teams are for work that benefits from multiple specialists. A typical team includes Atlas as the Manager plus agents for engineering, review, testing, writing, or research.

Team work happens across two main surfaces:

- **Team Hub**: the overview where you scan teams, unread updates, roster summaries, settings, and recent work.
- **Team Workspace**: the single-team control room where you chat, assign work, inspect coordination, review tasks, browse files, and preview the project.

## Team Hub

Use **Team Hub** before you enter a specific workspace. Team cards show the team name, working folder label, roster summary, agent status, unread or recent activity, settings access, and an open action.

When you select a team, the focused preview gives you the team roster, active status, focused inbox messages, and a primary action to open the team chat or workspace. Use this view when you need to compare several teams or check which one needs attention.

## The Manager

The Manager is the agent that coordinates a team. In most teams, this is Atlas. Instead of you assigning each step to each specialist, you give Atlas an outcome and it decides who should do what.

Atlas typically:

- Plans the work and breaks it into tasks.
- Delegates each task to the right specialist on the team.
- Follows up when an agent finishes or gets blocked.
- Routes inter-agent messages through **Agent Comms**.
- Creates **User Tasks** for decisions, approvals, blockers, or interruptions that need you.

Atlas hands work to specialists through dispatch messages. Specialists reply through **Agent Comms**, while your direct conversation stays in chat. See [chat and dispatch](/features/chat-and-dispatch) for how dispatches appear in the UI and how to watch them.

Every managed team has one Manager. How often Atlas asks before continuing is set by **Interruption Level**, described below. If you do not need a Manager, use a [solo agent](/features/solo-agents) instead.

## When to use a team

Use a team when:

- The work has multiple parts.
- You want Atlas to coordinate specialists.
- You need review or QA after implementation.
- Agents should hand work to each other.
- You want to monitor a longer workflow.

## Interruption level

Teams with Atlas can run under different interruption levels. Each level controls how often Atlas turns decisions, approvals, blockers, or scope changes into **User Tasks**.

| Level | Atlas can do without asking | Atlas asks before |
| --- | --- | --- |
| **Minimal** | Plan, delegate, follow up, and keep routine work moving | True blockers, missing access, high-risk work, or required approval |
| **Balanced** | Coordinate routine work and existing tasks | Meaningful new workstreams, unclear scope, medium or high risk, or completion approvals |
| **Hands-on** | Continue narrow follow-ups | Major assignments, broad follow-ups, risky decisions, and completion reviews |
| **Manual** | Wait for explicit approval on most coordination | Most delegations, follow-ups, re-routes, and new-work steps |

Start with **Balanced** for a new team. It keeps the team moving while giving you a chance to catch scope changes.

Older labels such as **Autopilot**, **Approve New Work**, and **Approve Each Step** map to the current interruption levels.

## Team workspace layout

The **Team Workspace** has three primary areas:

- The agent network, where Atlas and specialists show status, alerts, and coordination relationships.
- The right rail, where you use chat, **Agent Comms**, **User Tasks**, **Agent Tasks**, **Task Health**, **Assign Task**, and agent settings.
- The bottom tray, where you inspect **Activity**, **Files**, **Uploads**, and file previews.

When Atlas is selected, the right rail shows the manager operations tabs: **User Tasks**, **Agent Tasks**, and **Task Health**. Use these tabs for approvals, blockers, tracked work, and risk review instead of hunting through chat.

Use **Assign Task** when you want to create tracked work directly. You can choose the assignee, title, instructions, deliverable, done condition, acceptance criteria, phase, risk, whether completion needs human review, and whether to notify and start the agent.

Use the bottom tray when you need evidence for the work in progress. **Activity** shows recent tool, command, context, and message events. **Files** lets you browse the working folder and open previewable files. **Uploads** lists files attached through team chat.

Agent settings also open from the right rail. Use them to review identity, specialization, instructions, context settings, skills, and MCP server access for the selected agent.

## Hard pause

**Hard pause** stops in-flight team dispatches and blocks new automatic dispatch. Use the team settings pause control when a team is moving too quickly, a provider is misconfigured, or you want to inspect work before anything else runs. Resuming a paused team does not auto-restart agents. They run again when prompted or messaged.

Syndicate can also soft pause a team while an unresolved **User Task** waits for your input. You do not manually unlock a soft pause; reply to or resolve the relevant **User Task**.

## Project preview

Click **Preview** from the workspace header to open the team's project preview. Syndicate reuses an existing running server when it can. If none exists, it starts a default dev server from the team's working directory and opens the preview when a URL or port is available.

To stop or redirect managed work, tell Atlas what should stop, then cancel any specific task from **Agent Tasks**. If several active tasks should stop, use the board's abort-all-active option.
