Agents are specialized AI workers. Each agent has a name, specialization, provider model, instructions, optional skills, optional MCP server access, and context settings that control what it receives during runs. To add agents, install community packages, or create a custom agent, see [Agent Marketplace](/guides/agent-marketplace).

## Agent anatomy

| Area | What it controls |
| --- | --- |
| **Identity** | Agent name and specialization |
| **Soul** | The agent's role, behavior, boundaries, and working style |
| **Context** | History, tasks, inbox, references, team context, and token budget |
| **Skills** | Slash-invocable workflows available to the agent |
| **MCP Servers** | Installed MCP servers explicitly enabled for that agent |
| **Provider and model** | The provider model used when the agent runs |

## Specializations

Syndicate includes built-in specializations such as:

- Manager.
- Frontend Developer.
- Backend Developer.
- QA Engineer.
- DevOps Engineer.
- Technical Writer.
- Researcher.
- Finance Analyst.
- Data Scientist.
- Marketing Strategist.
- E-commerce Specialist.
- AI/ML Engineer.
- Mobile Developer.
- Game Developer.

Some areas split into more focused roles, such as UI Designer, Test Engineer, Site Reliability Engineer, AI Engineer, ML Engineer, and Security Engineer.

## Agent Marketplace

The Agent Marketplace is where Syndicate surfaces the catalog of available agent specializations. Browse it to pick a role when adding an agent or to compare focused variants such as AI Engineer and ML Engineer.

The Marketplace is a discovery surface, not a separate workspace. Anything you add from it lands as an agent inside the [team](/features/teams) or solo workspace you choose.

## Agent settings

Open **Agent Settings** from the right rail in **Team Workspace** or from a solo workspace.

Use it to:

- Rename non-manager agents.
- Review the agent's specialization.
- Read the agent's **Soul** instructions.
- Change context settings.
- Review skills and slash commands.
- Enable or disable MCP servers for that agent.
- Delete a non-manager agent after confirmation.

Manager names are locked.

## Soul

Every agent has instructions that define how it behaves. Syndicate calls these instructions the agent's **Soul**.

Good instructions are specific. Avoid generic traits like "be helpful" and prefer concrete rules such as "include file paths when reporting code changes."

## Context

Context settings determine how much prompt context the agent receives.

You can adjust:

- Preset: **Lean**, **Balanced**, or **Rich**.
- History lookback.
- Maximum characters per message.
- Role filters.
- Inbox lookback.
- Reference limits.
- Token budget.
- Whether to include team context, connectors, tasks, inbox, and references.

Prompt analytics can show estimated tokens, prompt size, history included, and section breakdowns.

## Skills and slash commands

Skills add domain-specific workflows. A Technical Writer might use documentation skills, while a QA Engineer might use test planning, browser testing, and regression-check skills. See [skills](/features/skills) for how to find and attach them.

When an agent has slash-invocable skills, the chat input can show a slash-command menu. Selecting a command inserts `/{skill-name}` into the message.

## References

References attach project files, docs, and notes to an agent so they show up as context during runs. Scope each reference to the agent that actually needs it. Fewer, sharper references usually produce better runs than dumping every file at every agent.

## MCP access

If MCP servers are installed in settings, enable specific servers per agent. Keep access scoped to the agents that actually need it. See [MCP servers](/features/mcp) for installation and configuration.
