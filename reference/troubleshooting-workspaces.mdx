This page covers team-side issues: Atlas behavior, dispatch, working folder, **User Tasks**, **Agent Comms**, and hard pause. For provider connection or auth problems, see [Troubleshooting](/reference/troubleshooting).

## Atlas is not delegating

Symptoms: you ask Atlas to do something, but no tasks appear and no specialists wake up.

Common causes:

- **Interruption Level** is **Hands-on** or **Manual**, and Atlas is waiting for your approval in **User Tasks**.
- The team is hard paused.
- Atlas asked you a question and the related **User Task** is still open.
- Atlas already delegated and is waiting on a specialist that is **Blocked**.

What to try:

1. Select Atlas and open **User Tasks**.
2. Resolve any pending decision, approval, blocker, or interruption prompt.
3. Confirm the team is not hard paused.
4. Open **Agent Tasks** and check for blocked or in-progress work.
5. Open **Agent Comms** for the relevant specialist.

## Agent is stuck on Blocked

Symptoms: an agent's status stays **Blocked** and no further work happens.

Common causes:

- The agent reported a problem it cannot solve alone, such as a missing reference, unclear requirement, failed tool call, or missing access.
- Atlas has not received the update yet.
- The blocker is waiting for a **User Task** response.

What to try:

1. Read the agent's latest chat and **Agent Comms** messages.
2. Open Atlas and check **User Tasks**.
3. Provide the missing input, reference, or decision.
4. Use **Nudge Manager** in **Task Health** if the blocked task needs Atlas attention.

## Automatic dispatch is not firing

Symptoms: a specialist sends a message, but no downstream agent wakes up.

Common causes:

- The target agent is not idle.
- The team is hard paused.
- A **User Task** is soft-pausing the team until you respond.
- Atlas is waiting for approval because of the current **Interruption Level**.

What to try:

1. Confirm the target agent is idle.
2. Confirm the team is not hard paused.
3. Resolve open **User Tasks**.
4. Send Atlas a clear next instruction, or manually message the target agent.

## Agent Comms is noisy

Symptoms: coordination messages pile up and the team feels hard to follow.

What to try:

1. Select Atlas and read **Agent Comms** first.
2. Open **User Tasks** and resolve anything that needs you.
3. Check **Task Health** for loops, blockers, or low-quality tasks.
4. Give Atlas one clear next instruction instead of replying to every specialist.
5. Raise **Interruption Level** if you want Atlas to ask more often.

## Working directory looks wrong

Symptoms: an agent edits the wrong files, cannot find files, or reports paths outside your project.

Common causes:

- The team is pointed at a different folder than you intended.
- The folder was moved, renamed, or deleted on disk.

What to try:

1. Open **Team Settings** and confirm the working directory.
2. If the folder moved, relink the team to the new path.
3. Stop active work before relinking.
4. Re-dispatch and confirm the agent now reads from the right location.

## Hard pause does not resume agents

Symptoms: you resumed a hard-paused team, but agents stay idle.

This is expected. Resuming a paused team only allows dispatch to run again. It does not restart agents automatically. Restart work by sending a message, manually messaging an agent, or giving Atlas a new instruction.

## Stop Team versus Stop

Use **Stop** in chat to cancel the current response for one agent.

Use **Stop Team** when the whole team needs a hard pause. **Stop Team** stops current team dispatches and blocks new automatic dispatch until you resume the team.

## Stopping runaway work

If a team is doing the wrong thing or moving too fast:

1. Click **Stop** for any active agent response doing the wrong work.
2. Click **Stop Team** if multiple agents are involved.
3. Review **Agent Tasks** and cancel work that should not continue.
4. Check **User Tasks** and **Task Health**.
5. Adjust instructions, references, or **Interruption Level** before resuming.

## Preview does not open

Symptoms: **Preview** does not open, opens the wrong project, or cannot find a running server.

What to try:

1. Confirm the team has the right working directory in **Team Settings**.
2. Confirm the project has a runnable dev server.
3. Use **Activity** to inspect recent commands and server output when available.
4. Stop and restart the preview if the server is stale.

## File preview does not load

Symptoms: **Files** opens, but a selected file does not preview.

What to try:

1. Confirm the file still exists in the working directory.
2. Click **Refresh** in **Files** if available.
3. Try a smaller text, code, or image file to confirm preview works.
4. Close the preview pane with **Escape** and reopen the file.

## MCP tool is configured but not available in a run

MCP credentials and tools are local to the machine and the provider runtime. If a server is configured in Syndicate but the active provider run does not expose it, the agent will say it is configured but unavailable in that run.

What to try:

1. Confirm the MCP server is enabled for the selected agent in **Agent Settings**.
2. Confirm the provider that ran is the one the server is wired to.
3. Reconnect the provider or reinstall the MCP server if it fails repeatedly.
