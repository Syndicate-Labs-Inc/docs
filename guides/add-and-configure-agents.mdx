Use this guide after you choose an agent in the [Agent Marketplace](/guides/agent-marketplace).

You can add agents to a team when you want Atlas to coordinate multiple specialists. You can create a solo agent when you want one focused specialist with no Manager.

## Before you start

Make sure you have:

- At least one connected provider.
- A team, unless you plan to create the solo agent from Marketplace.
- A clear job for the agent.
- Any references, skills, or MCP servers the agent needs.

If you have not connected a provider yet, follow [Connect providers](/guides/connect-providers).

## Choose the agent source

Open **Agent Marketplace**, then choose the source that fits your goal:

| Goal | Source |
| --- | --- |
| Use a proven Syndicate role | **Syndicate** |
| Use a package from another creator | **Community** |
| Use one of your saved custom agents | **User Made** |
| Create a new reusable agent | **Add Agent Package > Agent Maker** |

For source-specific setup, see [Built-in Syndicate agents](/guides/agent-marketplace/built-in-syndicate-agents), [Community agents](/guides/agent-marketplace/community-agents), or [Agent Maker](/guides/agent-marketplace/agent-maker).

## Add the agent to a workspace

1. Select the agent package in Marketplace.
2. Review the agent description, suggested workflow, and any setup notes.
3. Click **Add to...**.
4. Choose **Existing Team** or **Solo Workspace**, the option for a solo agent.
5. If you choose **Existing Team**, select the team.
6. If you choose **Solo Workspace**, name the solo agent and choose the project folder when prompted.

After you add the agent to a team, it appears in that team workspace. Atlas can assign it work through dispatch, and you can inspect its activity in **Agent Tasks**, **Agent Comms**, and chat.

After you create a solo agent, you work with that agent directly. There is no Manager and no team-level dispatch.

## Review the agent settings

Open the agent settings from the workspace and review:

| Setting | What to check |
| --- | --- |
| **Identity** | The agent name and specialization match the job you expect. |
| **Soul** | The role instructions describe responsibilities, boundaries, and output expectations. |
| **Provider and model** | The agent uses a connected provider and a model that fits the work. |
| **Team agent context** | The agent receives the right task, **Agent Comms**, references, and team context. |
| **Solo agent context** | The agent can use direct chat, the project folder, references, skills, and MCP server access. |
| **Skills** | Reusable workflows are attached only when they help the agent do repeatable work. |
| **MCP Servers** | External tool access is enabled only for the agents that need it. |

<Tip>
Start narrow. Add the smallest useful set of references, skills, and MCP servers, then expand only when the agent needs more context or tool access.
</Tip>

## Write useful role guidance

If you can edit the agent's **Soul** or package instructions, describe:

- What the agent owns.
- What it should avoid.
- Which files, systems, or references matter.
- How it should report progress.
- What a good final answer looks like.

Weak guidance:

```txt
Help with the project.
```

Better guidance:

```txt
Review onboarding docs for missing setup steps, unclear UI labels, broken links, and unsupported claims. Do not edit files unless asked. Return issues with file paths, impact, and suggested wording.
```

## Attach references

Attach references when the agent needs stable project knowledge, such as product notes, design files, API docs, or existing plans.

Use the narrowest useful scope:

- Workspace references for context every agent should see.
- Team references for context shared by one team.
- Agent references for specialist-only context.

For the full workflow, see [Attach references](/guides/attach-references).

## Enable skills and MCP servers

Use skills for repeatable workflows. Use MCP servers for external tools or data the agent must access.

Keep both scoped:

- Add skills that match the agent's recurring work.
- Enable MCP servers only for the agent that needs that access.
- Avoid giving broad external access to every agent in a team.

For setup steps, see [Use MCP servers and skills](/guides/use-mcp-and-skills).

## Verify the agent

Send a small request before relying on the agent for a larger task.

Good first prompts:

- "Read the project folder and summarize what you need to do this job well."
- "Review this plan and list the top five risks. Do not edit files."
- "Use your attached references to draft a checklist for this workflow."

Expected result: the agent responds in the right role, uses the expected provider, and asks for missing context instead of guessing.

For team agents, check that the agent receives the right task, **Agent Comms** messages, references, and team context. For solo agents, check direct chat, the project folder, references, skills, and MCP server access.

## Troubleshooting

| Symptom | What to check |
| --- | --- |
| The agent does not appear in the workspace | Reopen the workspace and confirm you selected the right destination under **Add to...**. |
| The agent cannot run | Check that its provider is connected and the selected model is available. |
| The agent misses obvious context | Review attached references and context settings. |
| The agent cannot use an external tool | Confirm the MCP server is installed and enabled for that agent. |
| A community agent reports a missing runtime | Reopen the package detail page and follow its setup notes. |
