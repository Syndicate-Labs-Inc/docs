A team is a [workspace](/features/workspaces) where Atlas coordinates specialists against one local project folder. Use this guide when you want a team to inspect, edit, test, review, or document a real project.

## Before you start

Make sure you have:

- Installed Syndicate and signed in.
- Connected at least one [provider](/guides/connect-providers).
- Picked the local project folder the team should use.

## Create the team

<Steps>
  <Step title="Start team creation">
    Click **Create Team**.
  </Step>
  <Step title="Choose a setup path">
    Choose the setup path that fits how much help you want:

    | Path | Use when |
    | --- | --- |
    | **Template** | You want a prebuilt team shape. |
    | **Guided description** | You want Syndicate to draft a roster from a project description. |
    | **Build Your Team** | You want to start with Atlas and add specialists yourself. |
  </Step>
  <Step title="Review the roster">
    Confirm Atlas and the specialists. Edit agent names, roles, or instructions before you create the team.
  </Step>
  <Step title="Create the team">
    Finish setup to open the new team workspace.
  </Step>
</Steps>

## Choose the working folder

The working folder is the local project folder the team can use during runs. Agents use it when they read files, edit code, run commands, and gather project context.

Choose the folder that contains the project you want the team to work on. Avoid selecting a parent folder that contains unrelated projects.

<Tip>
Start with one focused project folder. You can create another workspace later if you need a different project context.
</Tip>

## Pick an interruption level

**Interruption Level** decides how often Atlas asks before continuing.

| Level | Use when |
| --- | --- |
| **Minimal** | You trust Atlas to plan, delegate, and follow up without routine approval. |
| **Balanced** | You want Atlas to coordinate routine work, but ask before meaningful new work or higher-risk steps. |
| **Hands-on** | You want Atlas to ask before major assignments, broad follow-ups, risky decisions, and completion reviews. |
| **Manual** | You want to approve most delegations, re-routes, and follow-ups. |

Start with **Balanced** for a new team. It keeps the team moving while giving you a chance to catch scope changes.

## Send a first task

Keep the first task small. Ask Atlas to inspect the project before you assign implementation work.

For example:

```text
Inspect this project and summarize:
- What the app does.
- How to run it locally.
- Which files are most important.
- What you would improve first.
```

## Verify the team is working

After you send the task, check that:

- Atlas responds with a plan or dispatches a specialist.
- **Agent Tasks** shows tracked work when Atlas or you create a task.
- The assigned agent moves out of idle state.
- **Agent Comms** shows any inter-agent messages.
- **User Tasks** shows any decisions, approvals, or blockers that need your attention.

If nothing starts, check that the team has a connected provider and that the working folder is available.
