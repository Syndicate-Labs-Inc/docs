Create a team when work needs coordination across Atlas and specialists. A team is a workspace tied to one local project folder, with Atlas coordinating the specialists that work inside it.

Use [Quick start](/guides/create-your-first-team/quick-start) when you want the shortest path. Use this guide when you want to understand the setup choices before you create or revise a team.

## Choose a setup path

Syndicate gives you three ways to create a team.

| Path | Use when | What you decide |
| --- | --- | --- |
| **Template** | You want a proven team shape for a common workflow. | Pick the template, confirm the project folder, and review the generated roster. |
| **Guided description** | You know the project and outcome, but want Syndicate to suggest the roster. | Describe the project, review the proposed Atlas-led roster, then edit before creating. |
| **Build Your Team** | You know exactly which agents you want. | Start with Atlas, add specialists, and configure each role yourself. |

Start with **Template** when you are new to Syndicate. Use **Guided description** when the project has a clear goal but the team shape is not obvious. Use **Build Your Team** when you already know the roles you need.

## Pick the project folder

Every team needs a working folder. Agents use this folder when they inspect files, run commands, edit code, and gather project context.

Pick the smallest folder that contains the project. Avoid a parent folder that contains unrelated repositories, private notes, or files the team does not need.

## Review the roster

Before you create the team, review:

- **Atlas**: the coordinating Manager that plans, delegates, and follows up.
- **Specialists**: the agents that handle focused work, such as coding, review, writing, or research.
- **Names and roles**: labels that make handoffs and task ownership easy to scan.
- **Instructions**: constraints each agent should follow while working.
- **Provider and model choices**: the provider and model each agent uses for runs.

Keep the first roster small. You can add specialists later if the work needs more coverage.

## Choose an interruption level

**Interruption Level** decides when Atlas must ask before acting.

| Level | Best for | What to expect |
| --- | --- | --- |
| **Minimal** | Trusted teams and routine work. | Atlas plans, delegates, and follows up without routine approval. |
| **Balanced** | Most new teams. | Atlas coordinates current work, but asks before meaningful new work, unclear scope, or higher-risk steps. |
| **Hands-on** | Sensitive projects or first-time setup. | Atlas asks before major assignments, broad follow-ups, risky decisions, and completion reviews. |
| **Manual** | Highly controlled work. | Atlas asks before most delegations, follow-ups, and re-routes. |

Start with **Balanced** unless you already know the team and project are low risk.

## After creation

When the team opens:

1. Send one small inspection task to Atlas.
2. Check **Agent Tasks** for tracked work.
3. Check **Task Health** for blocked, risky, or under-specified work.
4. Watch whether a specialist moves out of idle state.
5. Open **Agent Comms** to inspect specialist handoffs.
6. Open **User Tasks** if the team asks for input.
7. Use **Activity**, **Files**, and **Uploads** in the bottom tray to inspect run events, project output, and attached files.
8. Click **Preview** if the working folder contains a runnable app.
9. Review the first result before assigning implementation work.
